新しい機能を追加するときは、このファイルを最初に読むこと。

## L-001: dedup要求 vs 問題空間サイズの不整合

### 何が起きたか
テストで `count: 100, types: ["multiplication"], digits: 1, allowDuplicates: false` を実行したところ、
generator が retry 上限に達して `生成失敗: 100回試行しても条件を満たす問題を作れませんでした` で例外終了した。

### 原因
1桁 × 1桁 の乗算は `9 × 9 = 81通り` しか存在しない。100問のユニーク生成は数学的に不可能。
division も同様に 81通りが上限。

### 教訓
- **問題空間サイズを意識せずに dedup + count を指定すると必ず事故る**
- retry枯渇エラーは「バグ」ではなく「設定矛盾」の検出結果でもあるので、
  将来的には generator 側で「このtypes+digitsでは最大N問しかユニーク生成できません」と
  事前チェックして警告する仕組みを入れるべき。
- テストで小さなドメインをテストするときは `allowDuplicates: true` を明示する。

### 予防ルール
- **新しい Problem プラグインを書くときは、「この設定で生成可能な最大ユニーク問題数」を頭の中で見積もる**。
- generator に `estimateMaxUnique(config)` のようなヘルパーを追加する検討（MVP後）。

## L-002: answer を文字列で持つと分数拡張で破綻する

### 何が起きたか
初期設計では `answer: "8"` と単純な文字列にしかけた。
設計レビュー段階で「分数が入った瞬間に `"1/2"` と `"0.5"` のどちらが正解か曖昧になる」と気付いた。

### 教訓
answer は必ず構造化して持つ。表示用文字列は `display` フィールドに分離する。

```js
// ❌ だめ
answer: "8"

// ✅ よい
answer: { type: "int",  value: 8,             display: "8" }
answer: { type: "frac", num: 1, den: 2,       display: "1/2" }
```

### 予防ルール
- **外に出るデータは必ず `{ type, ...構造化データ, display }` の3点セット**
- 表示は常に `display` を読む。値の比較・計算は `type` 別の分岐で構造化データを使う。
- 文字列を parse して値を取り戻すコードを書いたら**ただちに設計見直し**。

## L-003: question を文字列だけで持つと逆方向テストが書けない

### 何が起きたか
「問題 → eval → answer 一致」を確認する逆方向テストを書こうとしたとき、
`question: "3 + 5 = □"` という文字列しかないと再評価できないことに気付いた。
文字列パーサを書き足すと、そこが新たなバグの温床になる。

### 教訓
**内部表現はAST、表示時に文字列化、が鉄則。**

```js
{
  expr: { type: "op", op: "+", left: { type: "num", value: Frac(3) }, right: ... },
  question: "3 + 5 = □"  // render(expr) から生成
}
```

AST さえ持っておけば、`evalNode(p.expr)` で何度でも再評価でき、
逆方向テストが自然に書ける。

### 予防ルール
- 式に関わるデータ構造を作るときは、**必ず AST を先に設計する**。
- 文字列化は「出力の直前だけ」。パーサは絶対に書かない（MVPスコープでは）。

## L-004: Math.random() は seed 再現性を壊す

### 何が起きたか
未発生（設計段階で回避）。
だが要件定義（REQUIREMENTS.md 48-14）で明記されており、
「同じ設定で生成したのに毎回違う問題が出る → ユーザーが混乱」という事故が実際に報告されている。

### 教訓
- `Math.random()` は seed 指定不可。グローバルで一切使わない。
- SeedRandom (xorshift32) を全 Generator で共有し、呼び出し側が seed を注入する。

### 予防ルール
- **grep禁止語：`Math.random`**
- コードレビューで `Math.random` が入ったら即 reject。
- SeedRandom(0) は xorshift が詰むので、コンストラクタで `|| 1` フォールバック済み。

## L-005: Frac の約分忘れ／過剰約分

### 何が起きたか
未発生（設計段階で回避）。
REQUIREMENTS.md 48-7 に記載の典型事故。
`6/8` のまま返したり、符号処理を手動でやって `-1/-2` のような不正状態を作ったりする。

### 教訓
- Frac コンストラクタで**必ず** GCD で約分する。
- 符号は**必ず**分子に寄せる（分母は常に正）。
- ユーザー（＝他のコード）が `{ n: 6, d: 8 }` のような生の状態を作れないようにする。

### 予防ルール
- **Frac のフィールド `n`, `d` を外部から書き換えない（read-only扱い）**
- 生の分数を受け取る関数は必ず `new Frac(...)` でラップしてから使う。
- 比較は `.eq()` を使う。`p.n === q.n && p.d === q.d` はコンストラクタ保証があるので成立するが、将来的に TypeScript 化したら readonly 強制する。

## L-006: 減算で負数が出る

### 何が起きたか
未発生（設計段階で回避）。
`a - b` で `rng.int()` をそのまま使うと `a < b` のケースが必ず出る。
小学生向けプリントでは負数は原則NG（REQUIREMENTS.md 4-9）。

### 教訓
生成後にバリデーションで弾く（retry依存）のではなく、
**生成ロジック自体で「必ず非負になる形」で作る**。

```js
// ❌ 危険：後からバリデーション頼み
let a = rng.int(1, 99);
let b = rng.int(1, 99);
if (a - b < 0) return null; // retryで拾う

// ✅ 安全：生成時にスワップ
if (a < b) [a, b] = [b, a];
```

### 予防ルール
- 「制約を満たす問題の作り方」を**生成ロジックに埋め込む**。
- retry は最後の安全網。メインの制約は生成時に構造的に保証する。
- 他の保証も同じ発想で：
  - 割り切れる除算 → 商×除数で被除数を逆算
  - 逆算問題 → □の値を先に決めてから式を構築
  - 繰り上がりなし → 各位ごとに生成

## L-007: 浮動小数点を算数に使わない

### 何が起きたか
未発生（設計段階で回避）。
REQUIREMENTS.md 48 の最重要リスクの1つ。
`0.1 + 0.2 === 0.3` は `false`。小数問題で詰む。

### 教訓
- 数値は**全て Frac 経由**。`number` 型での加減乗除は禁止。
- 小数 `0.3` を扱うときは `new Frac(3, 10)` に変換してから計算する（将来実装）。

### 予防ルール
- **grep禁止パターン：`+`, `-`, `*`, `/` を `Frac` 以外の値に使うコード（算数ロジック部分）**
- rng が返す整数と、ループカウンタなどの制御用 `number` は除外。
- ツール化は MVP 後でよいが、レビュー時は人間が目視で確認する。

## L-008: retry は必ず上限を持つ

### 何が起きたか
設計段階で回避したが、初期案では `while (!isValid(p)) { p = generate(...) }` と
書きかけた。これは制約矛盾時に無限ループする。

### 教訓
- `while (true)` や上限なし retry は禁止。
- 必ず `MAX_RETRY = 100` で上限を持ち、超えたら例外を投げる。
- 例外メッセージには `count`, `types`, `digits` など設定情報を含めて、
  ユーザーが何を直せばいいか分かるようにする。

### 予防ルール
- **grep禁止パターン：`while (true)`**
- 全ての retry ループに `attempts < MAX` ガードを付ける。
- 上限超過は例外、`null` ではない（null だと呼び出し側で握りつぶされやすい）。

## L-009: テストで小さなドメインを扱うときの落とし穴

### 何が起きたか
L-001 の再掲になるが、テスト観点で言うと：
`generateSheet({ count: 100, types: ["multiplication"], digits: 1 })` を書いた瞬間、
デフォルトの `allowDuplicates: false` が効いて最初のテスト実行が落ちた。

### 教訓
- テストの目的を明確にする：
  - **ロジックの正しさ**を見たい → `allowDuplicates: true` で大量生成
  - **dedup の効き**を見たい → 問題空間が count より十分大きい設定にする
- この2つを混ぜない。

### 予防ルール
- テストケースのコメントに「何を確かめるテストか」を1行で書く。
- generator の呼び出しで `allowDuplicates` を省略したら、何が起きるかレビュー時に考える。

## L-010: プラグイン登録漏れ

### 何が起きたか
未発生（bootstrap.js に集約することで回避）。
だが「新しい Problem を追加 → `registry` に register し忘れ → unknown type エラー」は
プラグイン構造でよくある事故。

### 教訓
- **登録は1ファイルに集約する**（`js/bootstrap.js`）。
- 新しい問題を追加する手順を README に明記する：
  1. `js/problems/xxx.js` を作る
  2. `js/bootstrap.js` に import + register の2行を追加

- この手順から外れたら事故る、と諦める。

### 予防ルール
- bootstrap.js の先頭コメントに「**ここに追加しないと動かない**」と大書する（実施済み）。

# Phase 2 で追加された教訓

## L-011: answerMode の "auto" は問題ドメインを無視する

### 何が起きたか
分数問題 `4/5 + 1/4` の答えが `1.05` と**小数表記**で出た。
`4/5 + 1/4 = 21/20`、`21/20` は `d=20` で 10² の約数なので `fracToDecimalString` が
小数文字列を返せてしまい、`formatAnswer(auto)` がそれを採用した。

### 原因
`formatAnswer(auto)` のロジックは「小数で表せるなら小数を優先」だった。
これは**計算結果を見て**判断するだけで、**元の問題が何で構成されているか**を見ていない。
分数の問題には分数で答えるべき、という教育的コンテキストが抜け落ちていた。

### 教訓
`auto` モードは値の形を見るだけでは足りない。
**問題ドメイン（使っている数の種類）から、答えの表示形式を決める責務はプラグイン側にある。**

### 予防ルール
- プラグインの `generate()` 内で、`config.numberTypes` から `answerMode` を明示的に選ぶ：
  - `proper_fraction` を含む → `"fraction"`
  - `decimal1` を含む → `"decimal"`
  - それ以外 → `"int"`
- `formatAnswer(auto)` に頼るのは「プラグイン側が何も判断できない」特殊ケースのみ。
- テストで「分数問題の答えに `.` が含まれていないこと」を明示的にチェックする
  （`!/\.\d/.test(p.answer.display)`）。

## L-012: 同期テストハーネスは async 関数を sileng に通す

### 何が起きたか
Phase 2 で新テストを書くときに動的importを使って `async () => { ... }` を渡した。
ハーネスは `try { fn(); } catch { ... }` という同期構造だったため、
**async関数内のアサーション失敗が catch されず、全テストが "✓" で通ってしまう**可能性があった。

### 原因
- async関数はPromiseを返すだけなので、同期try/catchは例外を拾えない。
- `fn()` が即座にreturnし、`passed++` が実行される。実アサーションはマイクロタスクキューで後から走る。

### 教訓
- **テストハーネスと非同期関数は両立しない**。どちらかに統一する。
- MVP では全てのimportを静的にしてテストを同期で保つのが最もシンプル。

### 予防ルール
- テストは `function () { ... }` で書く（`async` 禁止）。
- 必要なimportは**ファイル先頭で静的に**行う。`await import(...)` は使わない。
- もし将来どうしても必要になったら、ハーネス側を `await fn()` に書き換えてから導入する。

## L-013: 多項式の中で除算位置を動かすのは難しい

### 何が起きたか
`arithmetic.js` で 3 項 `a ÷ b op c` や `a op b ÷ c` のような式をサポートしようとしたとき、
除算位置が先頭以外のときに「割り切れ保証」を入れる処理が複雑化した。
`(a + b) ÷ c` なら左側を再構築する必要があり、バグの温床になりそうだった。

### 教訓
**MVP では除算の位置を制限するのが正解**。
「全ての構文的バリエーション」を最初から目指すと、エッジケースで必ず破綻する。

### 予防ルール
- `arithmetic.js` の3項以上 × 除算は、**除算を最初の演算 (`i === 0`) の位置に限定**する。
- 除算が2番目以降の位置に出たら `null` を返して generator に retry させる。
- 将来的に「除算位置フリー」を実装する際は、左から累積評価しながら割り切れを逆算構築する
  専用ロジックを別ファイル（`DivisionBuilder.js` 等）に分離する。

## L-014: 小数・分数の×÷は生成が一気に難しくなる

### 何が起きたか
小数 × 小数 や 分数 ÷ 分数 の「きれいな答えになる」問題を自動生成しようとすると、
制約が一気に厳しくなる。例えば `0.3 × 0.4 = 0.12` は小数第1位を超えるため排除が必要。
`2/3 × 3/4 = 1/2` のように約分でうまくいくケースもあるが、事前に保証する方法が複雑。

### 教訓
Phase 2 では「小数・分数は加減算のみ」「×÷は整数のみ」という**安全弁**を置いた。
これにより生成成功率が高く保たれ、テストが安定した。

### 予防ルール
- `arithmetic.js` の中で「×÷を含む演算子列 → numberTypesを整数に強制」というガードを入れる（実装済み）。
- 小数×整数、分数×整数などの「片方が整数」のケースを次フェーズで段階的に追加する。
- 一度に全てを解禁しようとしない。

## L-015: テスト項目は「発生しうる問題」を先回りで書く

### 何が起きたか
L-011（分数問題に小数答えが出るバグ）を発見したのは**手動スモークテスト**だった。
自動テストでは「answerが正しく計算されているか」しか見ておらず、
**表示形式が問題ドメインと整合しているか**を検証していなかった。

### 教訓
「値が正しい」ことと「表示が正しい」ことは別のテスト項目。
両方を明示的に書かないと、片方が通って片方が破綻するバグは見逃される。

### 予防ルール
- 新しいプラグインを追加するときは、以下のテスト項目を必ずセットで書く：
  1. **計算の正しさ**：`evalNode(expr)` と `answer` が一致
  2. **制約の遵守**：非負・自明計算なし・割り切れ、など
  3. **表示形式の整合**：問題ドメインに応じた期待される形式（小数に `.` がある、分数に `/` がある等）
  4. **negative case**：分数問題に小数が混じっていないこと、等

## L-016: `hasTrivial` の初期定義が狭すぎた（実動作で発覚）

### 何が起きたか
論理テストは全部 green だったが、**実際に生成された問題を目視した**ところ、以下のような
「数学的には有効だが教材として無価値」な問題が混ざっていた：

- `73 ÷ 73 = 1`（`a ÷ a`）
- `5 ÷ 5 = 1`
- `1 × 7 = 7`（初期の hasTrivial は `×1` を見ていたが、Phase 1 プラグインは hasTrivial を呼んでいなかった）

初期の `hasTrivial` は `×0 ×1 ÷1` だけをチェックしていたため、
これらのパターンはすり抜けていた。

### 原因
- 「自明計算」の定義が狭かった。`a ÷ a = 1` や `a − a = 0` のような **「式の形は自明ではないが答えが自明に決まる」** ケースを想定していなかった。
- Phase 1 プラグインが `hasTrivial` を通さずに直接 `buildProblem` していた。新しいチェックを追加しても適用されなかった。

### 教訓
- 「自明」の境界は、**式の構造**だけでなく **答えの情報量** も見て判断する。
- 全プラグインが共通のバリデーションレイヤを通るように設計する。

### 予防ルール
- `hasTrivial` は以下を全て検出する（実装済み）：
  `×0`, `0×`, `×1`, `1×`, `÷1`, `÷0`, `+0`, `0+`, `−0`, `a÷a`, `a−a`
- **論理テストが通ってもスモークテストで必ず目視確認する**。100問くらい実際に印刷想定で眺めると、教材として違和感があるパターンが見えてくる。
- プラグインを増やすたびに `hasTrivial` を呼ぶのを忘れないこと。`buildProblem` の中に組み込むか、共通のバリデーションフックを作るのが次の改善。

## L-017: 「2桁」の意味が曖昧（÷の被除数問題）

### 何が起きたか
`digits: 2` で ÷ を生成したら `495 ÷ 99 = 5` のような**被除数が3桁**の式が出た。
「2桁の割り算」を指定したユーザーの期待は「両オペランドが2桁」だったのに、
実装は「除数は2桁、被除数 = 除数×商 で成り行き」だった。

### 原因
- 「digits=2」という設定値が何を指すのか、明確に定義していなかった。
- 除算は逆算構築なので、除数を 2桁 に固定すると、被除数は自動的に大きくなりがち。
- 仕様の抜け穴をコード側が気づかないまま「数学的には整合」と判定してしまった。

### 教訓
- 設定項目の「意味」は、コードを書く前に**1文で明文化**する。
- 「digits=2」 = 「全ての数値が2桁の範囲 [10, 99] に収まる」と定義するなら、
  除算では **q を `qMax = floor(max/d)` で制限する**必要がある。
- 単純な範囲指定でも、演算子によって制約が変わる。

### 予防ルール
- 実装済み：`arithmetic.js` の除算は `qMin = max(2, ceil(min/d))`, `qMax = floor(max/d)` で被除数を
  digits の範囲内に収める。範囲内で成立しないときは `null` を返して retry。
- 今後「digits」のような範囲系オプションを追加するときは、**README の config 表に「何を指すか」を明記する**。
- 仕様の曖昧さを見つけたら、まず文章化 → 次に実装、の順を守る。

## L-018: 論理テストだけでは不十分、実動作確認が必須

### 何が起きたか
47 tests green の状態で zip を出したが、**実際の出力を眺めるまで L-016, L-017 のバグに気づかなかった**。
論理テストは「こうあるべき」の検証であり、「こう見える」の検証ではなかった。

### 教訓
テストは2層で考える：
1. **論理テスト**（assert ベース）：値の正しさ、制約の遵守、再現性
2. **実動作テスト**（目視＋スクリプト）：典型シナリオで実際に生成して眺める

両方を通して初めて「使える」と言える。論理テスト 100% green でも、
実動作の違和感は別ルートで検出する必要がある。

### 予防ルール
- プロジェクトに `smoke.mjs` と `verify.mjs` を常備する（本リポジトリでは `scripts/` 配下に置いた）。
- リリース前（＝zip化前）に必ず以下を実行：
  ```bash
  node js/tests/test.js    # 論理テスト
  node scripts/smoke.mjs   # 典型シナリオ目視
  node scripts/verify.mjs  # 深掘り検証
  node scripts/stress.mjs  # 大量＋時間計測
  ```
- 出力を眺めて違和感があったら、その違和感を lessons に言語化して fix する。
- 「テストが通ったから完成」は嘘。**人が使って納得したから完成**。

## L-019: ES モジュールは `file://` 経由で読めない（生成ボタンが無反応バグ）

### 何が起きたか
ユーザーから「生成ボタンが機能しない」と報告を受けた。
ブラウザで `index.html` をダブルクリックで開いた場合、**何も起きずボタンが無反応**だった。

### 原因
Phase 2 までの `index.html` は以下のように ES モジュールとしてロジックを import していた：

```html
<script type="module">
  import { generateSheet } from "./js/generator.js";
  // ...
</script>
```

ブラウザのセキュリティモデルでは、**`file://` プロトコル経由で ES モジュールをロードすると CORS エラーで必ず失敗する**。
開発者のコンソールを見ていないユーザーには「ボタンが無反応」としか見えない。

私（Claude）は README で「簡易サーバー経由で動かしてください」と書いたが、
**ユーザーは普通にダブルクリックで試す**のが自然な挙動。そこが UX の抜け穴になっていた。

### 教訓
- 配布物の HTML は、**何もインストールせずダブルクリックで動く**のがデフォルトの期待。
- ES モジュールは開発体験としては綺麗だが、「ユーザーが配布ファイルをそのまま開く」シナリオでは致命的に相性が悪い。
- 「簡易サーバーを立ててください」は**仕様ではなく事故**である。README に書いてあっても事故は事故。

### 予防ルール
- 最終配布版の `index.html` は**完全自己完結**にする：
  - すべてのロジックを `<script>` タグ内に inline
  - `import` / `export` / `type="module"` を一切使わない
  - 外部 `.js` ファイルへの参照をゼロにする
- モジュール版（`js/` 配下）は Node.js テストと開発用として別途維持する。
- **リリース前チェックリストに「index.html をダブルクリックで開いて生成ボタンが反応するか」を追加**する。
- 検証手順の具体例：
  ```bash
  # インライン版とモジュール版の出力が一致することを自動比較
  node /tmp/inline_logic_test.mjs
  ```
- `grep -n 'type="module"\|import.*from\|export' index.html` で何も出ないことを確認する。

## L-020: UI の責務分割（タブ構造）

### 何が起きたか
Phase 2 までの UI は「設定フォーム → 生成ボタン → 同じ画面に問題と解答を並べて表示」という
1画面構成だった。ユーザーからの要求で「問題生成 / 問題表示・印刷 / 解答」の3タブ構造に分離した。

### 教訓
算数プリントという**教材の性質**を考えると、UI は以下の3段階に明確に分かれるべき：

| 段階 | 目的 | 表示されるべきもの |
|---|---|---|
| 生成 | 設定を決める | フォームのみ。問題も解答も見えない |
| 問題 | 生徒に見せる・印刷する | 問題のみ。解答は絶対に見えない |
| 解答 | 丸つけに使う | 解答のみ |

特に「問題を生徒に見せるときに解答が視界に入る」のは致命的なUX。

### 予防ルール
- プリント系のアプリでは、**問題と解答を同じ画面に並べない**。タブで物理的に分離する。
- 印刷機能は「今見えているタブだけ」を対象にする（`@media print` で他のタブは隠す）。
- タブの並び順は使用順と一致させる：① 生成 → ② 表示・印刷 → ③ 解答。
- 初回状態ではタブ2・3を disabled にしておき、問題が生成されて初めて有効化する
  （空のタブを開かれても困惑するだけなので）。

## L-021: 【BUG】解答タブが裸の値だけを表示していた

### 何が起きたか
v3 の解答タブは、問題 `7/8 + 2/3 = □` に対して **`37/24` という値だけ**を表示していた。
ユーザーから「答えに□はいらない」と指摘を受けて、本来の意図が判明した：
**解答タブは「□を答えで埋めた完成式」を表示すべき**だった。

```
[v3 までの誤った表示]              [本来あるべき表示]
(1) 37/24                          (1) 7/8 + 2/3 = 37/24
(2) 31/24                          (2) 2/3 + 5/8 = 31/24
(3) 73/63                          (3) 4/9 + 5/7 = 73/63
```

### 原因
- 答えのデータ構造を `{ type, value, display }` と設計したとき、`display` を「答えの値の文字列」と
  解釈したまま、UI 側でもそのまま貼っていた。
- 「丸つけのとき、生徒の解答用紙の問題番号と先生の解答リストを照合する」というワークフローを
  考慮していなかった。問題の式を併記しないと、答えだけ並べられても先生がどの問題か分からない。
- ユーザーの指摘「答えに□はいらない」は、最初は意味が分からなかった（既に□は表示していなかった）。
  実際は **「答えの行を、□を埋めた完成式の形にしてほしい」** という意味だった。

### 教訓
- 「答え」というデータには2つの概念がある：
  1. **答えの値**（`37/24`）
  2. **答え合わせ用の完成式**（`7/8 + 2/3 = 37/24`）
- どちらが必要かは UI の文脈（誰が・何のために見るか）で決まる。
  解答タブは「先生が丸つけに使う」ので (2) が必要だった。
- ユーザーの指摘文を文字通りに解釈できないときは、**実際にユーザーが何の作業をしようとしているか**を
  想像しないと意味が取れない。

### 予防ルール
- 教材アプリでは「解答」は必ず **問題の式とセット** で表示する（`buildAnsweredLine(p)` ヘルパー）。
- データ構造の `display` フィールドは「値の文字列表現」のみを保持し、
  「文脈に応じた表示形」は UI 側のヘルパーで組み立てる。
- ユーザーフィードバックの解釈に詰まったら、**そのフィードバックを発した状況（どの画面で・何をしようとして）**
  を再現してから読み直す。

## L-022: 【UX】"a/b" 平文表記は小学生には読めない

### 何が起きたか
分数の答えを `37/24` のように **スラッシュ区切りの平文** で表示していた。
小学生（特に低学年）にとってこの表記は読みにくく、`37/24` を `37` 割る `24` と
誤解する可能性があった。教材として致命的。

### 原因
- 開発者の感覚で「数学的に正しければよい」と思っていた。
- LaTeX や MathJax は依存追加が大きいので避けたが、**CSS だけで分数表示は実装できる**
  ことに気づくまで時間がかかった。

### 教訓
分数は **分子と分母を縦に並べ、横棒で区切る視覚表現** にする。
LaTeX 不要、外部依存ゼロ、純 CSS の `display: inline-flex` で実装可能：

```html
<span class="frac">
  <span class="f-n">7</span>
  <span class="f-d">8</span>
</span>
```

```css
.frac {
  display: inline-flex;
  flex-direction: column;
  align-items: center;
  vertical-align: -0.55em;
  font-size: 0.85em;
}
.frac .f-n { padding: 0 6px 2px; border-bottom: 1.6px solid currentColor; }
.frac .f-d { padding: 2px 6px 0; }
```

### 予防ルール
- 教材アプリで **数式テキストを直接 innerHTML する前に**、必ず以下を通す：
  - `renderMath(str)`：`a/b` パターンを視覚分数 HTML に変換
  - 整数・小数・演算子はそのまま通過
- 視覚分数化のテスト項目を持つ：
  - `1/2 + 1/3 = □` → 分数 HTML が2つ含まれる
  - `3 + 5 = 8`     → 分数 HTML が含まれない
  - `3.4 + 1.2 = 4.6` → 分数 HTML が含まれない（小数点はスラッシュではない）
- 将来的に帯分数（`1と2/3`）や仮分数→帯分数変換を入れるときは、
  `renderMath` の責務を「文字列パターン → HTML」から「Frac構造体 → HTML」に
  リファクタする（パターンマッチでは帯分数を扱えないため）。
- LaTeX/MathJax に頼る前に、**単純なCSSで足りないかを必ず先に検討する**。
  追加依存ゼロが理想。

## L-023: 【BUG】手書きテストなのに問題文に `□` と `あまり` が印刷されていた

### 何が起きたか
ユーザーから直接の報告：

> ちがう。手書きで書き込むテストなので回答を書いてもダメだし、□を書いてもだめ

問題タブの印刷出力に以下のような文字列が出ていた：
- `3 + 5 = □`
- `17 ÷ 5 = □ あまり □`

紙に手書きで答えを書くテストプリントなので、`□` という記号自体が「ここに何か書く」という
過剰な誘導であり、`あまり` というラベルも答えのヒントになってしまう。
**問題文は `<expression> =` で終わって、あとは余白だけ**であるべきだった。

### 原因
1. **教材としての配布形態を意識せずにデータ構造を決めた**。
   `question: "3 + 5 = □"` という文字列を内部表現として使っていた。
   `□` をプレースホルダ記号として再利用すれば「答えを差し込む位置の目印」として
   解答タブで便利、という発想だったが、その内部表現が**そのまま印刷面に流れた**。
2. 「問題文＝印刷される文字列」と「ASTから生成される表示用テキスト」を区別していなかった。
3. 解答タブで `buildAnsweredLine` が `□` を answer.display で `replace` する作りになっており、
   `□` が必須の前提構造になっていた。

### 教訓
- **印刷物のデータは、印刷した時に「そのまま教材として配れる」状態であるべき**。
  内部都合のプレースホルダや目印を残さない。
- 問題文には絶対に出してはいけない情報を、明示的にリスト化する：
  - 答えの数値そのもの
  - `□` `_____` などの記入欄記号
  - `あまり` `答え` などのヒントになるラベル
  - 解法のヒントになる中間値
- 解答との突き合わせは、**問題文の構造を変えずに答えを「足す」だけ**で済むように設計する。
  `replace(/□/, ...)` ではなく `question + " " + answer.display`。

### 予防ルール
- **問題文の不変条件**（仕様化済み・テストで担保）：
  1. 問題文は `<expression> =` で終わる（末尾が `=`）
  2. `□` を含まない
  3. `あまり` `⋯` `_` `?` などの「答えの位置を示す記号やラベル」を含まない
  4. 答えそのもの（answer.display と同一の文字列）が問題文に現れない
- 上記をテストで自動チェックする（`js/tests/test.js` の Phase 3 セクション）：
  ```js
  assert.ok(!p.question.includes("□"));
  assert.ok(!p.question.includes("あまり"));
  assert.ok(!p.question.includes("⋯"));
  assert.ok(p.question.endsWith("="));
  ```
- **新しい問題タイプ（穴埋め・選択肢など）を追加するときは、問題文の不変条件を再定義する**。
  穴埋め問題なら「□が出てよい位置」が定義されるはずだが、それは別タイプの別仕様であり、
  通常の記述式とは混ざらないようにする。
- リリース前チェックリストに「印刷タブを開いて、答えのヒントになるものが映っていないか目視」を追加。

### 関連
- L-002（answerは構造化）：similar 設計ミス。表示と内部表現を混同しないこと。
- L-020（UI責務分割）：問題と解答を別タブにしたのは正解だが、問題文自体に答え位置が
  漏れていては意味がない。

## L-024: 【BUG】arithmetic プラグインが numberTypes をサイレントに整数へ落としていた

### 何が起きたか
`generateSheet({types:["arithmetic"], numberTypes:["decimal1"], ops:["*"], ...})` を実行すると、
ユーザーは「小数のかけ算」を期待しているのに、実際は `7 × 4 = 28` のような整数問題しか出てこなかった。
分数 (`proper_fraction`) + `["*"]` も同様に整数化されていた。

### 原因
`arithmetic.js` 内の MVP安全弁：
```js
const hasDivOrMul = opSeq.some(o => o === "/" || o === "*");
const effectiveTypes = hasDivOrMul ? ["integer"] : numberTypes;
```
×÷を含む小数・分数のきれいな結果生成は L-014 で「次フェーズの課題」とした。
が、その間の挙動を「サイレントに整数へ落とす」にしてしまったため、
ユーザーから見ると「指定したのに無視される」という最悪の不一致になっていた。

### 教訓
- **未対応の組み合わせは、サイレントに別物を返すのではなく、明示的に throw する**。
- 「動くけど期待と違う」は「動かない」よりはるかに見つけにくく、信頼を壊す。
- 安全弁は「ユーザーから見えない劣化」ではなく「ユーザーから見える失敗」にする。

### 予防ルール
- プラグインが config の一部を内部的に書き換える場合、必ずテストで
  「指定 → 出力」の整合性を assert する（L-026参照）。
- 「未対応の組み合わせ」は config 検証段階で fatal error として throw し、
  generator のリトライで握り潰されないようにする（`err.fatal = true`）。

## L-025: 【BUG】division プラグインが digits 引数を完全に無視していた

### 何が起きたか
`generateSheet({types:["division"], digits:2})` を実行しても、`9 ÷ 3` のような

```js
const divisor  = rng.int(2, 9);
```

## L-026: 【テスト設計】「存在チェック」を「不変条件チェック」に置換せよ

### 何が起きたか
v5 のテストには次のようなものがあった：
```js
test("括弧は結果が変わるときしか生成されない", () => {
  // ...
  let withParens = 0;
  for (const p of problems) if (p.question.includes("(")) withParens++;
  assert.ok(withParens > 0, "no parenthesized problems generated");
});
```
テスト名は強い不変条件を主張しているが、実際は「括弧付き問題が0件ではない」しか確認していない。
名前と中身の落差が、L-024 / L-025 と同じ「テストが緑なのに仕様違反」を許す土壌になっていた。

### 教訓
- **テスト名が主張する不変条件を、本当にコードで検証しているか確認する**。
- 「存在する」だけのテストは、ロジックが死んでも検出できない（カバレッジは取れるが意味がない）。
- 「ユーザー指定 → 出力」の整合性は、smoke テストや reverse-eval テストでは検出できない別レイヤである。

### 予防ルール
- テスト層を 4層に整理する：
  - **L1: 単体（純関数）** — Frac, Evaluator, helpers の入出力
  - **L2: プラグイン契約** — 全プラグインが満たす共通契約（reverse-eval, 不変条件）
  - **L3: 入出力整合**（新設） — 「指定した config が出力に反映されているか」
  - **L4: プロパティ / 統合** — 大量生成、決定性、混合 types
- L3 を厚くする：
  - `numberTypes:["decimal1"]` → 全オペランド display に `.` を含む
  - `digits:2` → 全オペランドが [10..99]
  - `ops:["+"]` → expr 内に `+` 以外の op が出ない
  - `notation:"amari"` → answer.display に "あまり" が出る
- 「N > 0」と書きたくなったら立ち止まり、「全 N について何が成り立つべきか」を書く。

### 関連
- L-018（論理テストだけでは不十分）：UI 側の話だったが、ロジック側にも同じ盲点があった。
- L-024 / L-025：このテスト設計の弱さが原因で発見が遅れたバグ。

## L-027: 【検証で発覚】L-025修正で digits=1 division の問題空間が縮みすぎた

### 何が起きたか
L-025 修正で「`division` プラグインは `digits` を尊重する」を実装し、
「`digits=1` → 被除数 ∈ [1..9]」という厳密な意味にした。
ユニットテストはすべて green になったが、smoke スクリプトを実走すると：
```
Phase1除算 九九範囲 seed=5
   1. 8 ÷ 2 = → 4
   2. 8 ÷ 4 = → 2
   3. 8 ÷ 4 = → 2
   4. 8 ÷ 4 = → 2
   5. 8 ÷ 4 = → 2
   6. 8 ÷ 4 = → 2
```
6問中5問が同じ問題に。被除数候補が `{4,6,8,9}` の4種しかなく、
`allowDuplicates:true` で同じ問題が連続して出てしまっていた。

### 原因
**「digits の意味」を取り違えた**。
小学校カリキュラム的には「1桁の割り算」は「九九を使う割り算」のことであり、
`24÷3` (被除数2桁) も含まれる。「被除数が1桁」という機械的解釈ではない。

| digits | 一般的解釈 (curriculum) | 機械的解釈 (誤) |
|--------|------------------------|------------------|
| 1      | 九九ベース、除数も商も 1桁 (被除数 ≤81) | 被除数 ∈ [1..9] |
| 2      | 被除数 2桁 ÷ 1桁 (例: 56÷7) | 被除数 ∈ [10..99] |
| 3      | 被除数 3桁 ÷ 1桁 (例: 384÷6) | 同上 [100..999] |

L-025 修正版は機械的解釈で実装してしまっていた。

### 教訓
- **「テストが全部 green」と「実動作が良い」は別物**。L-018 で同じことを書いたのに繰り返した。
- ユニットテストは「ルールを満たすか」しか見ない。「意味のある分布が出ているか」は実データを目で見ないと分からない。
- 修正後は必ず smoke / stress スクリプトを走らせ、出力を**目で**読むこと。

### 予防ルール
- バグ修正のあとは：
  1. ユニットテスト（`npm test`）
  2. **smoke スクリプト実走 + 出力目視**
  3. **stress スクリプト実走 + 出力目視**
  4. 仕様変更を伴うなら lessons.md 追記
  の4点セットをやる。
- 「桁数」「範囲」「個数」などの**意味**を持つ引数を実装するときは、
  実際にそのドメインの専門家（小学校算数の場合は教科書）の慣習を確認する。
- セマンティクス選択を伴う修正は、**テストの assert にもセマンティクスを書き込む**：
  ```js
  // 単に範囲チェックではなく、ドメイン固有の不変条件を書く
  test("digits=1: 九九範囲 (除数 [2..9]・商 [2..9]・被除数 ≤ 81)", ...)
  ```

### 原因
`arithmetic.js` 内：
```js

```

- **生成器のオプションは「最大値」だけでなく「固定値」も用意する**。

## L-029: 答えの折り返しが「= の前」ではなく途中で起きていた

### 何が起きたか
v9 まで `.problem` / `.answer-item` を flex 化した結果、長い式や狭い列幅で
折り返しが発生したとき、`=` が行末に取り残される or 中途半端な位置で改行されていた。
日本の算数教科書では、複数行になる式は `=` を**新しい行の先頭**に置くのが標準。
ユーザーから「段を変えるときはイコールから記載すること、縦に順番に記載する」と指示。

### 原因
- 行内のすべての要素 (フラクション、演算子、`=`) を 1 つの flex container に並べていた。
- `flex-wrap: wrap` で wrap させていたが、wrap 位置を制御していなかったので、
  `=` の前後どちらでも壊れる可能性があった。
- そもそも「答えの式は常に縦に並べる」という組版習慣を意識していなかった。

### 教訓
- **数式の表示は単に「inline でくっつけて wrap させる」ではなく、組版習慣を尊重する**。
  - 複数行になる場合、`=` は次の行の先頭に置く。
  - 答え行は「式」「= 答え」の 2 行構成にすると採点者が読みやすい。
- レイアウト変更は CSS だけでなく HTML 構造も含めて検討する。
  flex の wrap 制御は CSS 単独だと限界があり、明示的な break 要素 (`<i class="br">`) が必要。

### 予防ルール
- 数式表示の panel では、`=` の前に強制 line-break (flex の `flex-basis: 100%` を持つ
  ダミー要素) を必ず挿入する。
- これによって：
  - 1 行目: 式 (例: `1/2 + 1/3`)
  - 2 行目: `= 5/6`
  という「縦に順番に並ぶ」レイアウトが常に得られる。
- 第 2 行は `(1)` 番号の下ではなく式と同じ位置から始まるよう、
  `(1)` と式は別の wrap container に分ける。

### 関連
- L-022 (分数の視覚化): UI レイヤの組版判断は L-022 で一度議論したのに、
  本件で改めて踏み直した。

  「組版の常識を毎回見落とす」を防ぐため。

## L-033: L-024 fatal を粒度を上げて部分的に解放した (小数/分数 × 整数 を許容)

## L-036: 繰り上がり / 繰り下がり の制御

### 実装
`config.carry`: `"any"` (default) / `"with"` / `"without"`
- 整数の加減算ノードに対して column-wise で carry/borrow を判定
- AST 全体を walk し、`with` ならどこかに 1 つ以上、`without` なら 1 つも無い、を満たすかチェック
- 満たさなければ `null` 返却 → generator がリトライ
- 加減ノードが 1 つも無ければ `null` (判定不能)
- × / ÷ ノードはカウント対象外
- 非整数の `numberTypes` と組合せたら fatal (carry の概念が定義されない)

### ヘルパ
```js
additionHasCarry(a, b)    // 桁ごとに sum>=10 が一度でも起きるか
subtractionHasBorrow(a, b)// 桁ごとに 上 < 下 が一度でも起きるか (a>=b 前提)
```

### UI
「繰り上がり / 繰り下がり」select を追加。`指定なし / なし / あり` の3択。

### テスト (8件追加、計102)
- 1桁 + carry without: 全問 `a+b<10`
- 1桁 + carry with: 全問 `a+b>=10`
- 2桁 − carry without: 単位桁の borrow なし
- 2桁 − carry with: 全問で borrow が起こる
- 2桁 + carry without: 列ごと sum<10
- 3項 +/- carry without: 各 +/- ノードで違反なし
- fatal: carry + 非整数
- fatal: carry に未知の値

sync_check に 5 シナリオ追加 (21/21 一致)。

### 実機確認
```
1桁加算 carry なし:  2+1=3  1+8=9  4+5=9
1桁加算 carry あり:  7+5=12 5+8=13 9+6=15
2桁減算 borrow あり: 63-44=19 73-14=59
2桁減算 borrow なし: 96-73=23 87-77=10
```

### 教訓
- 「繰り上がり」は curriculum 的にも明確に概念が固まっているので、
  実装の判定基準も column-wise でクリアに書ける。
- 制約フィルタは「生成 → 検証 → reject」パターンが綺麗に決まる。
  特殊な生成器を作るより、汎用ジェネレータ + フィルタの組合せのほうが小さい。

### 関連
- L-026 (L3 入出力整合テスト): config 指定が出力に反映されているかを assert する方針
- L-024 / L-033: fatal で「未対応の組み合わせ」を明示する方針

## L-038: 穴埋め (□ プレースホルダ) L1 — 逆算問題の最小実装

### スコープ
SAKUMON_SECTIONS Ⅷ-1 / Ⅷ-2 (L1) を対象。
- 2項のみ
- 整数のみ
- 4 種の slot: `left` / `right` / `result` / `random`
- L2 (カッコの塊) / L3 (玉ねぎ) / L4 (複雑な分母) は次イテレーション

### 実装方針
**新しい AST node 型は作らない**。代わりに、既存の arithmetic プラグインで通常通り
`a op b` を生成し、後段で「どのスロットを□にするか」を選んで question 文字列だけ
書き換える。answer はそのスロットの値。

```
1. arithmetic 通常生成 → tree (a + b など) と problem を得る
2. placeholder !== "none" なら slot を決定
3. 元の display 文字列を組み立て直し:
   - left   → "□ + ${b.display} = ${result.display}"  answer=a
   - right  → "${a.display} + □ = ${result.display}"  answer=b
   - result → "${a.display} + ${b.display} = □"       answer=result
4. expr は元の tree のまま保持 (debug 用)、answer は新しい値で上書き
```

これにより:
- 既存の生成器の制約 (allNonNegative / hasTrivial / 割り切れ保証) が
  そのまま継承される
- 新しい AST node 型を増やさず、評価器・renderer・sync_check に変更ゼロ
- 「Ⅷ-2 非対称トラップ」(`15 - □ = 8` の答えは 7、誤答 23 を誘発) も
  値の substitution だけで自然に出る

### 制約 (今回 fatal にしている)
- `terms !== 2` → fatal「L1 は 2項のみ」
- `numberTypes` に非整数 → fatal「L1 は整数のみ」
- `placeholder` の値が未知 → fatal

### テスト (9 件追加 → 計 111)
- L3: left + 加算 → answer = c-b (逆算が正しい)
- L3: right + 減算 → answer = a-b (非対称、誤答 a+b ではない)
- L3: right + 除算 → answer = a/b (非対称)
- L3: left + 乗算 → answer = c/b
- L3: result + 全演算 → answer = eval(expr)
- L3: random → 3 slot 全てが出現、□ が必ず 1 個
- fatal: 3項
- fatal: 小数
- fatal: 不正値

### sync_check
6 シナリオ追加 (placeholder 4 種 + random + fatal)。27/27 一致。

### 実機確認
```
□ + 5 = 8      →  3
□ + 8 = 17     →  9
76 − □ = 18    →  58   (75−18=58、76+18=94 ではない)
8 ÷ □ = 2      →  4    (8÷2=4、8×2=16 ではない)
4 × 4 = □      →  16
5 × □ = 40     →  8
```

### renderMath との相性
`□` は単一 Unicode 文字で、renderMath の token 走査には特別扱いがない
(分数 / 帯分数 / 演算子に該当しない)。プレーンテキストとしてそのまま出力され、
`= □` も既存の eq-group に正しく入る (確認済)。CSS 変更不要。

### 教訓
- **既存基盤を活かす最小実装** が L1 の正解。新 node 型を作らず substitution 1 段で
  解禁できた。
- 「Ⅷ-2 非対称の答え方」は子どもにとっては大きな概念だが、ジェネレータ側では
  「元式が a-b=c なら answer=b」と書くだけ。**知識のある側 (生成器) と知識のない側 (生徒) で
  非対称性のコストが違う** ことを再認識した。
- L2/L3 (3 項以上、カッコの塊) は AST walk と「外側からの剥がし順」の概念が必要に
  なるので、L1 と質的に違う設計判断が出てくる。次回に分けて正解。

### SAKUMON_SECTIONS への効果
✅ 解禁:
- Ⅷ-1 (L1: □ が左辺)
- Ⅷ-2 (L1: □ が右辺・非対称)
- Ⅷ-7 (□ の配置制御 — left/right/result/random)

🟡 部分対応 (L2/L3 は未):
- Ⅷ-3, Ⅷ-4, Ⅷ-5, Ⅷ-6

副次的に解禁の見込み (UI 側で組合せれば):
- Ⅰ-11 10 の補数 (`□ + 3 = 10` が出せる)
- Ⅰ-12 100/1000 の補数 (digits を上げれば)
- Ⅲ-6 あまりあり穴埋め (remainder プラグインに同じ仕組みを足せば → 別途)
- Ⅴ-19 分数補数 (placeholder + 分数を解禁する次イテレで)

### 関連
- L-013: ÷ 位置制限。2項では問題にならない (位置 0 = 唯一)
- L-026: 入出力整合テスト方針が L-038 でも生きている
- L-036: filter 系 (carry) と placeholder は直交。同時指定可能

## L-039: 小数第2位 (decimal2) — 桁数不一致系セクション解禁の前提

### スコープ
SAKUMON Ⅳ-1〜Ⅳ-3 の解禁を目的に、operand として `decimal2` を追加。
×÷ の解禁は別イテレ (今回は L-024/L-033 の制約を維持したまま)。

### 実装
1. `_numberGen.js` に `genDecimal2(rng, intMax)` を追加
   - 整数部 0..intMax、小数部は 2 桁、第2位は **必ず非ゼロ** (`1.20` のような decimal1 もどきを避ける)
   - 内部値は分母 100 の Frac
2. `genNumber` の dispatch に `case "decimal2"` を追加
3. `arithmetic.js` の answerMode auto を `decimal1 || decimal2` に拡張
4. inline bundle に同じ実装をミラー
5. UI チップに「小数 (2位)」を追加 (既存「小数」は「小数 (1位)」にラベル更新)

### 制約 (今回維持している L-024/L-033 の境界)
- decimal2 のみ + × → fatal (decimal × decimal は未対応)
- decimal2 のみ + ÷ → fatal (÷ は依然未対応)
- decimal2 + integer + × → OK (L-033 と同じ smart sampling)

### テスト (6 件追加 → 計 117)
- L1: genDecimal2 の値構造 (分母100、第2位非ゼロ)
- L3: decimal2 +- → 全オペランドが分母100 の約数
- L3: decimal1 + decimal2 混合 → 桁数不一致が出る (Ⅳ-2)
- L3: 整数 + decimal2 → 答えが正しい (Ⅳ-3)
- fatal: decimal2 のみ + × (L-024 維持)
- L3: 整数 × decimal2 (L-033 互換)

### sync_check
4 シナリオ追加 (decimal2 +-、decimal1+decimal2 混合、整数+decimal2 -、整数×decimal2)。31/31 一致。

### 実機確認
```
decimal2 +-:        7.31 − 4.07 = 3.24    1.11 + 6.64 = 7.75
decimal1 + decimal2: 5.9 − 1.17 = 4.73    5.84 − 0.63 = 5.21
整数 + decimal2:     8 − 4.37 = 3.63      6 − 5.72 = 0.28
整数 × decimal2:     6 × 0.12 = 0.72      4 × 3.39 = 13.56
```

### 教訓
- **新しい数の種類を足すコストは小さい**: ジェネレータの dispatch 1 行 + 関数 1 個 + answerMode の OR 拡張だけ。
  Frac が分母を保持しているおかげで、加減算は混合しても自然に動く。
- **「末位非ゼロ」制約** がないと、ジェネレータは `1.20` を作って後段の表示が `1.2` (decimal1 と区別不能) になり、
  「decimal2 を選んだのに decimal1 が出た」と感じられる。生成段で防ぐのが正解。
- L-024/L-033 の境界は **数の種類を増やすたびに自動的に正しく適用される** (`hasNonInt` チェックは
  `t !== "integer"` で判定するので、新しい型が即座に「非整数」扱いになる)。
  これは L-033 を整理した時の設計判断 (per-type ではなく per-int) が効いている。

### SAKUMON への効果
✅ 解禁:
- Ⅳ-1 (既存。decimal1 で十分)
- **Ⅳ-2** (桁数不一致 — `5.9 − 1.17` のような問題が出るように)
- **Ⅳ-3** (整数と小数の加減 — `8 − 4.37` のような問題が出るように)

🟡 部分的に届く:
- **Ⅳ-4** (`1.25 × 4 = 5`) — 整数 × decimal2 の経路で出るが「.00 を削る」可視化は無い
  (内部的には formatAnswer が末尾ゼロを既に削っているので、表示としては `5` で正しい)

❌ まだ届かない (× decimal × decimal や ÷ 系が必要):
- Ⅳ-5〜Ⅳ-10、Ⅳ-11〜Ⅳ-13、Ⅳ-14
- 次イテレで L-014/L-024 の境界を更に押し下げる必要がある

### 関連

- L-024: fatal の境界

## L-040: 印刷レイアウト (A4縦・10問/枚・縦項番)

## L-041 改訂: 1 ページ問題数を増やす (8/10/18 → 10/20/30)

### 何が起きたか

実際の市販計算ドリルや「100マス計算」(10×10=100 問/枚) を見れば自明で、
**教科書の文字サイズ規格に引きずられて保守的になりすぎていた**。

### 検算
A4 縦 (297mm) − 30mm 余白 − 25mm ヘッダ ≒ 242mm の縦スペースに対し:
- 16pt × 10 行 = 16pt × 1.4 × 10 = 224pt ≈ 79mm (中、ゆったり)
- 12pt × 10 行 = 12pt × 1.4 × 10 = 168pt ≈ 59mm (小、余裕)
- 22pt × 5 行 = 22pt × 1.4 × 5 = 154pt ≈ 54mm (大、余裕)

ヘッダ + 上下余白を引いた残りが 240mm 近くあるので、いずれも余裕で収まる。
**前回の 8/10/18 は実装上の余裕を取りすぎ**。

| 小 | 12pt | 3×10 | 18 → **30 問** |

### 教訓
- **「教科書本文の規格」と「計算ドリルの実用密度」は別物**。教科書は読解前提なので

## L-042: 印刷プレビューが白紙になる (window.print() の非同期性)

### 何が起きたか
v22〜v25 で印刷ボタンに以下のコードを書いていた:
```js
$("tab-print").classList.add("print-target");
window.print();
setTimeout(() => $("tab-print").classList.remove("print-target"), 200);
```
ユーザーから「印刷画面に何も表示されていない」と報告。

### 原因
1. **`window.print()` は Chrome では同期ブロックせず即座に return する**。
   200ms 後の `remove()` がプレビュー描画より先に走り、CSS の
   `.tab-panel { display: none !important; }` が効いて白紙になる。
2. **ユーザーがブラウザの Ctrl+P を直接押した場合は `print-target` が一度も付かない**。
   この経路でも白紙になる。

### 修正
W3C 標準の `beforeprint` / `afterprint` イベントを使う:
```js
window.addEventListener("beforeprint", () => {
  const active = document.querySelector(".tab-panel.active");
  if (active && (active.id === "tab-print" || active.id === "tab-answer")) {
    active.classList.add("print-target");
  }
});
window.addEventListener("afterprint", () => {
  document.querySelectorAll(".tab-panel").forEach(p => p.classList.remove("print-target"));
});
$("printBtn").addEventListener("click", () => window.print());
```
これで in-app ボタンでも、ブラウザ直接 Ctrl+P でも、active タブが正しく印刷される。

### 教訓
- **`window.print()` の同期/非同期はブラウザ依存**。setTimeout で剥がす方式は競合状態。
  必ず `beforeprint` / `afterprint` を使う。
- **印刷経路は in-app ボタンだけでない**。ユーザーは Ctrl+P / ⌘+P / メニュー → 印刷など
  ブラウザネイティブの経路を使うことが多い。これらでも動く設計にする。
- **印刷バグは画面で見ても気付けない** (画面は normal 表示)。実際に印刷プレビューを
  開いて確認するまで完了としない (P-001 / L-018 / L-027 系)。
- 古い WebKit (wkhtmltoimage) でのレンダリング検証は Modern browser のバグ再現には
  不十分。検証は必ず Chrome/Firefox の実印刷プレビューで行う。

## L-043: 列バランス崩れ — 生成数 < 1ページ容量 のとき右列が空になる

### 何が起きたか
v27 で 中プリセット (40問/ページ・2列×20行) にしたあと、ユーザーが
**20 問だけ生成**して印刷プレビューを見たところ、20 問全部が **左列に縦に並び**
右列が完全に空白になった。さらに 16 問でブラウザの自動改ページが入り 1/2 ページに
分かれてしまった。

### 原因
CSS で `grid-template-rows: repeat(20, 1fr); grid-auto-flow: column;` と
**固定 20 行**にしていた。column-major flow なので、items 1-20 が 1 列目を
埋め尽くし (20 行使い切る)、2 列目が空のまま。仕様上正しい挙動だが
「20 問なら 10 行 × 2 列にバランスしてほしい」というユーザー期待と乖離。

### 修正
CSS の固定行数は撤廃しないが、JS 側で**そのページの実問題数に応じて**
`<div class="problems" style="grid-template-rows: repeat(N, 1fr)">` を
inline で上書きする。`N = Math.ceil(itemsOnPage / cols)`。
- 20 問 / 2 列 → 10 行 × 2 列 ✓
- 40 問 / 2 列 → 20 行 × 2 列 ✓ (フル時)
- 7 問 / 2 列 → 4 行 × 2 列 (右列の最後が空) ✓
- 60 問 / 3 列 → 20 行 × 3 列 ✓ (フル時)

`PRINT_PRESETS` に `cols` を追加し、`renderPrintPanel` / `renderAnswerPanel`
両方で同じロジックを使う。

### 教訓
- **「1ページ容量」と「実問題数」は別物**。CSS で容量側を固定するなら、
  JS で実数に応じた上書きが必要。grid-auto-flow:column と固定行数の組合せは
  「常にフル」を仮定した設計で、半端な数で破綻する。
- **実機確認は「典型ケース」だけでなく「フルでないケース」も**。L-041 系の
  改訂はずっと「20 問でフルに 20 マス埋まる」前提で考えていた。ユーザーが
  別の数を生成すると一気に崩れる。
- L-019 系: CSS と JS の二重実装。CSS 単独で完結しないなら JS を併用する
  必要がある (今回は inline style で上書き)。

### 関連
- L-040 / L-041 (再々改訂): 印刷レイアウトの歴史
- L-018 / L-027: 実画面確認まで完了としない (典型ケース以外も見る)

## L-044: `1fr` 行が効かない — 親に definite height がないとフォールバック

### 何が起きたか
v28 で列バランス (L-043) を修正したあと、ユーザーから「下段の空白は改善しない」と指摘。
20 問が 10 行 × 2 列に並ぶようにはなったが、その 10 行が紙の上半分に詰まって
下半分が空白のまま。

### 原因
CSS Grid の `grid-template-rows: repeat(N, 1fr)` は **親が definite height** のときだけ
等分される。`.print-page` には高さを与えていなかったので、`1fr` は indefinite case の
`auto` (= max-content) にフォールバックし、各行が**問題の自然高**になっていた。
結果、行は等分されず紙の上半分に詰まった。

これは v25 引継ぎ時点の HANDOFF.md に「仮説 A 最有力」として書かれていた現象で、
L-042 (印刷バグ) → L-043 (列バランス) を直したことでようやく表面化した。

### 修正
`@media print` 内で:
```css
.print-page {
  height: calc(297mm - 30mm); /* A4 縦 - @page margin × 2 */
  display: flex;
  flex-direction: column;
  box-sizing: border-box;
}
.print-page .problems,
.print-page .answers {
  flex: 1 1 auto;
  min-height: 0;  /* flex item が grid を内包する時の縮小許可 */
}
```
親 `.print-page` に definite height、flex column で `.problems` を残り高フル占有
させる。これで `1fr` 行が正しく解決され、N 行が紙の縦全体に等分される。

### 教訓
- **`1fr` は親の高さに依存する**。CSS Grid の `fr` は flex の `fr` と違い、
  親が indefinite height のときは `auto` フォールバックする。これは仕様。
  動かない時は親に高さを与えるか、flex で残り高を渡す。
- **`min-height: 0` を忘れない**。flex item の中に grid/scroll を入れるとき、
  `min-height: auto` がデフォルトで内容より縮まないため、flex: 1 が効かない罠。
- **多段バグは順番に剥がれる**。L-042 (白紙) → L-043 (列バランス) → L-044 (1fr)
  と、表層のバグを直すたびに次の層が見えた。HANDOFF.md の「仮説 A」が
  ここでようやく的中。**仮説リストに書いた未検証項目は、表層が直ったら必ず再検証**する。

### 関連
- L-040 / L-041 (再々改訂) / L-042 / L-043: 印刷レイアウトの一連
- v25 HANDOFF.md「仮説 A」: 今回これが顕在化

## L-045: SAKUMON Ⅰ-2/Ⅰ-3/Ⅱ-1〜Ⅱ-3 — 補数 filter と 九九段別 filter

### 経緯
SAKUMON_VERIFICATION で 🟡 になっていた小規模 filter のうち、最も出題価値が高い5件
(Ⅰ-2 補数2 / Ⅰ-3 補数3以上 / Ⅱ-1〜3 九九段別) を 1 イテレーションで解禁する。

### 設計判断
1. **`complement` config キーを追加**: `"any" | "le2" | "ge3"`
   - `le2` = 補数 2 以下 = max(a,b) ∈ {8,9} かつ a+b ≥ 10
   - `ge3` = 補数 3 以上 = max(a,b) ≤ 7 かつ a+b ≥ 10
   - validation: `digits=1` かつ `ops=['+']` のときのみ有効。違反は fatal。
2. **`multiplicandSet` config キーを追加**: 1〜9 の整数の非空配列
   - 例: `[2,3,5]` で「九九 低段」。片方が set 内、もう片方は [1..9] 全域、位置はランダム。
   - validation: `digits=1` かつ `ops` に `*` を含むときのみ有効。違反は fatal。
   - **legacy `multiplication` プラグインにも対応**: ジェネレータ側で直接片方を set から選ぶ
     (post-gen filter ではなく pre-gen 制約) ので生成効率が最大。
3. **arithmetic 経路は post-gen filter**: 既存の carry filter と同じ層に追加。N項にも自然に拡張可能。

### Ⅰ-3 仕様の解釈について (記録)
SAKUMON_SECTIONS.md の Ⅰ-3 例には 8+4 が混じっているが、本実装では:
- 「補数 ≤ 2」= max(a,b) ∈ {8, 9} → 8+4 は **Ⅰ-2** に分類される
- 「補数 ≥ 3」= max(a,b) ≤ 7 → 7+5 は **Ⅰ-3** に分類される
仕様の例示に揺れがあると判断し、実装は「補数の値で機械的に分類」する方針を採用。
将来 Ⅰ-3 の境界を「max ≤ 7 の and/or 補数 ≥ 3」に変更する場合は本 lesson を更新。

- `index.html` に `<select id="complement">` と `<select id="multiplicandSet">` を追加 (carry の隣)
- multiplicandSet は CSV 文字列 `"2,3,5"` で持ち、`readUIConfig` で配列に変換

## L-046 (v31): filter 拡張 第二弾 — `carry='all'` / `forceZeroDigit` / `factor10` / `remainderClass`

### きっかけ
L-045 の手法 (狙い撃ち config を 1 イテレーションでまとめる) が高効率だったので、🟡/⬜ で残っていた filter 系 7 セクション (Ⅰ-4, Ⅰ-8, Ⅰ-9, Ⅱ-6, Ⅲ-3, Ⅲ-4, Ⅲ-11) を一気に解禁する。

### 実装
1. **`carry: "all"`** (連続両桁繰り上がり/下がり, Ⅰ-4 / Ⅰ-8)
   - post-gen filter として `additionAllColsCarry(a,b)` / `subtractionAllColsBorrow(a,b)` を新設
   - **carry_in/borrow_in 込みの筆算判定**が必要 (素朴な digit 比較ではダメ)
   - validation: digits=1 では fatal
2. **`forceZeroDigit: boolean`** (空位 0, Ⅰ-9)
   - `intHasZeroDigit(n)` + `treeHasZeroDigitOperand(node)` ヘルパー
   - 少なくとも一つのオペランドに 0 桁を含むことを要求 (post-gen filter)
3. **`factor10: boolean`** — arithmetic と remainder で**別実装**
   - arithmetic (Ⅱ-6): 全 × ノードで片方が 10 の倍数 (`treeAllMulFactor10`)
   - remainder (Ⅲ-11): 除数 a・被除数 b・あまり r をすべて ×10 にスケール
4. **`remainderClass: "any" / "large" / "max"`** (Ⅲ-3 / Ⅲ-4)
   - large = `r ≥ ⌈b/2⌉`、max = `r = b−1`
   - 既存 `remainderGenerate` を全面書換

### バグと修正 (このイテレーションでの試行錯誤)
1. **`subtractionAllColsBorrow` の素朴実装** (`while (bi > 0)` で全桁判定) → b の全桁が borrow を要求してしまい本来の意味とズレ。さらに 2 桁同士で「両桁 borrow」は a > b と矛盾するため**数学的に不可能**だが、retry 200 回で枯渇するだけでテストが赤くなって発覚
2. **`additionAllColsCarry` も carry_in 未考慮** だった。47+58 のような「ones 桁 carry → tens 桁が carry_in 込みで carry」が判定漏れ
3. **修正**: 両ヘルパーを「**下位 2 桁で連続して carry/borrow が発生 (carry_in/borrow_in 込み)**」という正しい筆算判定にリライト
4. **テストも合わせて修正**: subtraction-all テストは digits=3 に変更、assertion は緩める (`digit_a ≤ digit_b` で borrow_in 込みの borrow を許容)

### 反省 / 学び
